## Phân Tích 2 Vấn Đề Phát Hiện

### Vấn đề 1: HARD vẫn SLOW (~25 giây)

25 giây vẫn quá chậm để chơi game. Nguyên nhân:

`SortCandidates` được gọi tại **mỗi node trong cây đệ quy**, không chỉ ở root.

```
AI_FindBestMove()        → SortCandidates (root, ~40 candidates)
  Minimax() depth=5      → SortCandidates (~35 candidates)
    Minimax() depth=4    → SortCandidates (~30 candidates)
      Minimax() depth=3  → SortCandidates (~25 candidates)
        ...
```

Chi phí sort O(n log n) nhân với hàng triệu nodes = overhead đáng kể.
Giảm 35x nodes nhưng mỗi node lại tốn thêm chi phí sort → net gain ~11x thay vì 35x.

**Hướng cải thiện:** Chỉ sort ở các tầng depth cao (gần root), bỏ qua ở tầng depth=1,2.

### Vấn đề 2: pruneRatio vẫn thấp (5–8%) — đây là vấn đề đo lường

Metric hiện tại đang đếm sai:
```cpp
// Đang đếm:
s_nodesPruned++   // chỉ tăng 1 khi break xảy ra

// Nhưng mỗi lần break, thực tế có thể bỏ qua 20-30 candidates tiếp theo
// → 1 "prune event" = có thể tiết kiệm 25 nodes
```

Ví dụ: 30 candidates, break ở i=5:
- `nodesVisited` = 6, `nodesPruned` = 1
- pruneRatio được tính = 1/7 = 14%
- Nhưng thực tế bỏ qua 24/30 = **80% candidates**

**Kết luận:** pruneRatio 5–8% KHÔNG có nghĩa là Alpha-Beta kém hiệu quả.
Số nodes giảm 35x mới là con số phản ánh đúng hiệu quả thực tế.

**Hướng cải thiện metric:** Đo `candidatesSkipped` thay vì `breakEvents`.

## Bài Học Kỹ Thuật

**Move ordering hoạt động — nhưng có cost ẩn:**
- Sorting at every node adds O(n log n) overhead per node
- Tradeoff: ít nodes hơn nhưng mỗi node đắt hơn
- Optimal: sort deep nodes ít hơn hoặc không sort

**Metric thiết kế quan trọng:**
- Một metric tốt phải đo đúng thứ mình muốn đo
- `pruneRatio` hiện tại đo "break events", không đo "candidates skipped"
- → Cần redesign metric hoặc thêm `candidatesSkipped` counter (DR-004 tiềm năng)
